# Claudefy v1.4.0 — Implementation Design

**Date:** 2026-03-15
**Status:** Approved (post-review revision 1)
**Source:** `product-evolve.md` (31 issues from Opus desktop audit of v1.3.4)
**Approach:** Module-grouped waves (9 waves, all issues)

---

## Overview

All 31 issues from the PRD have been validated against the codebase. Every factual claim is accurate. This design organizes the implementation into 9 waves grouped by module area, minimizing file thrashing and ensuring dependencies flow naturally.

**Wave ordering is sequential and mandatory.** Each wave must be completed before the next begins. Waves 3 and 6 both modify `push.ts` and `pull.ts` — sequential ordering prevents merge conflicts.

## Wave 2: Lockfile (P0-03)

**New files:**
- `src/lockfile/lockfile.ts`
- `tests/lockfile/lockfile.test.ts`

**Modified files:**
- `src/commands/push.ts`, `pull.ts`, `override.ts`, `init.ts`, `join.ts`, `restore.ts`

### Design

```ts
interface LockInfo {
  pid: number;
  command: string;
  startedAt: string; // ISO timestamp
}

const LOCK_PATH = join(claudefyDir, ".lock");
const MAX_LOCK_AGE_MS = 10 * 60 * 1000; // 10 minutes
```

**Acquisition logic:**
1. No lock exists → write `{ pid, command, startedAt }` using `writeFileSync` with `wx` flag (O_CREAT | O_EXCL for atomic creation), return lock
2. Lock exists, PID dead → remove stale lock, re-acquire
3. Lock exists, lock expired (>10min) → remove, re-acquire
4. Lock exists, PID alive, lock fresh → blocked

**TOCTOU note:** Two processes could race to detect and remove a stale lock. Using the `wx` flag on the write step ensures only one succeeds — the loser gets an EEXIST error and retries the read-check cycle once. This is acceptable for a CLI tool.

**When blocked:**
- Quiet mode (hooks): exit code 0, no output
- Interactive: print message, exit code 0

**Integration — `withLock()` wrapper:**

```ts
export async function withLock(
  command: string,
  quiet: boolean,
  fn: () => Promise<void>,
): Promise<void> {
  const lock = Lockfile.tryAcquire(command, quiet);
  if (!lock) return; // blocked — tryAcquire already printed message or exited silently
  try {
    await fn();
  } finally {
    lock.release();
  }
}
```

Note: Return type is `Promise<void>` (not generic `Promise<T>`) because all command `execute()` methods return void. Callers do not use the return value.

**Mutating commands (require lock):** `push`, `pull`, `override`, `init`, `join`, `restore`

**Non-mutating commands (no lock):** `status`, `link`, `machines`, `doctor`, `config`, `hooks`, `logs`, `diff`

### Tests

- Acquire succeeds when no lock exists
- Acquire fails when another live PID holds it
- Stale lock (dead PID) auto-cleaned and re-acquired
- Expired lock (>10min) auto-cleaned and re-acquired
- Lock removed after `release()`
- Lock removed even if wrapped function throws
- Quiet mode: blocked → returns without output
- Interactive: blocked → prints message
- Atomic creation: concurrent `tryAcquire` calls → only one succeeds (test with `wx` flag behavior)

## Wave 3: Pull/Push Pipeline Fixes (P0-04, P1-05, P1-08)

### P0-04: Signal handler fix

**File:** `src/commands/pull.ts`

Replace `process.exit(1)` with flag-based interruption:

```ts
let interrupted = false;

const onSignal = (signal: string) => {
  interrupted = true;
  // Re-register for second signal → force kill
  process.once(signal, () => process.exit(128 + (signal === "SIGINT" ? 2 : 15)));
};

process.once("SIGINT", () => onSignal("SIGINT"));
process.once("SIGTERM", () => onSignal("SIGTERM"));
```

Note: The `.once()` listener auto-removes after firing, so there is no need to call `process.removeListener()` inside the handler. The `finally` block should still remove listeners as a safety measure using a named reference to the outer wrapper if needed, but the `.once()` guarantee makes the `removeListener` call in the current `finally` block a safe no-op.

**Interrupt checkpoints** — check `interrupted` at these points in the pull pipeline:
1. After pulling git changes (before decryption)
2. After decryption (before path remapping)
3. After path remapping (before settings merge)
4. Inside the file copy loop (between individual file copies to `~/.claude/`)

Second signal = force kill. Standard Unix CLI behavior.

**Result:** `finally` block now executes on signal. Exit codes: 130 (SIGINT), 143 (SIGTERM).

### P1-05: Merger array dedup fix

**File:** `src/merger/merger.ts`
**New test file:** `tests/merger/array-merge.test.ts`

Replace `if (!key) return source;` with three-tier strategy:

1. **Primitive arrays** (strings, numbers): set-based union using strict equality (not `String()` coercion). Remote order first, append local-only items.
2. **Object arrays with key** (`name`/`id`/`key`): existing key-based dedup (unchanged).
3. **Object arrays without key**: JSON-stringify-based dedup. Preserves both sides.

```ts
arrayMerge: (target: unknown[], source: unknown[]) => {
  // Primitive arrays: strict-equality union
  if (source.length > 0 && typeof source[0] !== "object") {
    const seen = new Set(source);
    const localOnly = target.filter(item => !seen.has(item));
    return [...source, ...localOnly];
  }

  // Object arrays: try key-based dedup
  const key = this.findArrayKey(source);
  if (key) {
    const remoteKeys = new Set(
      source.map((item) => (item as Record<string, unknown>)[key]),
    );
    const localOnly = target.filter(
      (item) => !remoteKeys.has((item as Record<string, unknown>)[key]),
    );
    return [...source, ...localOnly];
  }

  // Object arrays without identifiable key: JSON-based dedup
  const sourceJson = new Set(source.map(item => JSON.stringify(item)));
  const localOnly = target.filter(item => !sourceJson.has(JSON.stringify(item)));
  return [...source, ...localOnly];
},
```

Note: Uses `new Set(source)` with native `Set.has()` for primitives, which uses SameValueZero comparison (strict equality). This means `1` and `"1"` are treated as different elements, preserving type fidelity.

**Tests:**
- `["a", "b"]` + `["b", "c"]` → `["b", "c", "a"]`
- `["a"]` + `["a"]` → `["a"]`
- `[1, 2]` + `[2, "2"]` → `[2, "2", 1]` (type-aware, `1` ≠ `"1"`)
- `[{name:"x"}]` + `[{name:"y"}]` → key-based dedup works
- `[{foo:"bar"}]` + `[{baz:"qux"}]` → both preserved
- Empty arrays handled
- All existing merger tests pass

**Files modified:**

### Config

```ts
// src/config/types.ts — update encryption interface
encryption: {
  enabled: boolean;
  useKeychain: boolean;
  cacheDuration: string;
  mode?: "reactive" | "full"; // NEW — defaults to "reactive" for backward compat
}

// src/config/defaults.ts
encryption: {
  enabled: false,
  useKeychain: false,
  cacheDuration: "1h",
  mode: "reactive",

```

### Behavior matrix

| `enabled` | `mode` | Scanner hit | Result | Note |
|---|---|---|---|---|
| `false` | n/a | No | Plaintext | |
| `false` | n/a | Yes | Push blocked | |
| `true` | `"reactive"` | No | Allowlisted: plain. Unknown: encrypted. | **Existing behavior, preserved** |
| `true` | `"reactive"` | Yes | Scanner-hit + unknown: encrypted. Rest: plain. | **Existing behavior, preserved** |
| `true` | `"full"` | n/a | Everything encrypted | **New behavior** |

Row 3 and 4 describe the existing v1.3.4 behavior, not new logic. The `"reactive"` code path must not be refactored — only the `"full"` branch is new code.

### Implementation

In `push.ts`, when `mode === "full"`:
- Collect ALL files in both `configDir` and `unknownDir` into `filesToEncrypt`
- Scanner still runs regardless of mode. In `"full"` mode its results are informational. In `"reactive"` mode they drive encryption decisions. The `skipSecretScan` flag skips the scanner in both modes — in `"full"` mode this is fine (everything is encrypted anyway).

### Doctor addition

Replace the existing `"encryption"` check with an expanded version:

```ts
{
  name: "encryption",  // keep existing name to avoid breaking tooling
  status: config.encryption.enabled ? "pass" : "warn",
  detail: config.encryption.enabled
    ? `Enabled (mode: ${config.encryption.mode ?? "reactive"})`
    : "Disabled — files stored in plaintext",
}
```

### Migration

None. Missing `mode` field defaults to `"reactive"` — identical to current behavior. `types.ts` uses optional (`mode?:`) for backward compat with existing configs.

### Tests

- `mode: "reactive"` (default) — existing behavior preserved (only scanner-hit + unknown encrypted)
- `mode: "full"` — all files encrypted
- Missing `mode` in config → treated as `"reactive"`
- `mode: "full"` + `skipSecretScan: true` → all files encrypted, scanner skipped
- `mode: "reactive"` + `skipSecretScan: true` → only unknown-tier encrypted
- Doctor reports mode correctly

## Wave 5: Secret Scanner (P1-02, P2-08)

**Files modified:**
- `src/secret-scanner/scanner.ts`
- `src/config/defaults.ts`
- `src/config/types.ts` — add `secretScanner` config type
- `tests/secret-scanner/scanner.test.ts`

### P1-02: Regex fixes

**Twilio:** `/SK[0-9a-fA-F]{32}/` → `/\bSK[0-9a-fA-F]{32}\b/`

Keep case-insensitive hex (Twilio docs show uppercase hex in SIDs). Only add word boundaries to prevent matching inside longer hex strings.

**Datadog:** `/dd[a-z]{0,2}_[0-9a-zA-Z]{32,}/` → `/\b(?:ddapi|ddog|dd)_[0-9a-zA-Z]{32,}\b/`

### P2-08: Custom patterns

**Config:**
```ts
secretScanner: {
  customPatterns: [], // Array<{ name: string; regex: string; flags?: string }>
},
```

Note: `flags` field added to support case-insensitive matching and other regex flags. Defaults to no flags if omitted.

**Scanner constructor accepts custom patterns:**
```ts
constructor(customPatterns: Array<{ name: string; regex: string; flags?: string }> = []) {
  const custom = customPatterns.map(p => {
    try {
      return { name: p.name, regex: new RegExp(p.regex, p.flags) };
    } catch (e) {
      throw new Error(`Invalid regex in custom pattern "${p.name}": ${(e as Error).message}`);
    }
  });
  this.patterns = [...BUILTIN_PATTERNS, ...custom];
}
```

**Config schema validation (Wave 8):** When Wave 8 adds schema validation, `secretScanner.customPatterns` must be included in the schema map to avoid "unknown key" warnings.

### Tests

- Twilio: `SK` + 32 lowercase hex chars matches ✓
- Twilio: `SK` + 32 uppercase hex chars matches ✓ (uppercase hex is valid)
- Twilio: `xSK` + 32 hex chars does NOT match (no word boundary)
- Twilio: Inside a SHA-256 hash → no match (boundary prevents)
- Datadog: `ddapi_abc123...` matches ✓
- Datadog: `dd_x` does NOT match (too short)
- Custom pattern: `{ name: "Internal", regex: "MYCO_[A-Za-z0-9]{32}" }` detects match
- Custom pattern with flags: `{ name: "Case", regex: "secret_key", flags: "i" }` matches `SECRET_KEY`
- Invalid custom regex → error with pattern name
- All existing tests pass (regression)

## Wave 6: ~/.claude.json Selective Sync (P1-01)

**New files:**
- `src/claude-json-sync/claude-json-sync.ts`
- `tests/claude-json-sync/claude-json-sync.test.ts`

**Modified files:**
- `src/commands/push.ts`
- `src/commands/pull.ts`
- `src/config/defaults.ts`
- `src/config/types.ts` — add `claudeJson` config type

### Security Model

**Critical context:** The existing pull pipeline strips `mcpServers` from `settings.json` (in `DANGEROUS_KEYS`) because remote settings could inject malicious MCP servers that execute arbitrary code. Wave 6 introduces a separate sync path for MCP servers via `~/.claude.json`. This requires an explicit security model.

**Security approach: User-controlled trust with opt-in confirmation.**

1. `claudeJson.sync` defaults to `true` but MCP server sync requires **explicit opt-in via `claudeJson.syncMcpServers: true`** (defaults to `false`).
2. On first pull that contains remote MCP servers with `syncMcpServers` disabled, print a one-time notice:
   ```
   Remote store contains MCP server configurations.
   To sync MCP servers, run: claudefy config set claudeJson.syncMcpServers true
   Warning: MCP servers can execute arbitrary commands. Only enable if you trust all machines sharing this store.
   ```
3. When `syncMcpServers` is enabled, MCP server configs pulled from remote are written to `~/.claude.json` — the user has accepted the trust model.
4. The `settings.json` stripping of `mcpServers` via `DANGEROUS_KEYS` remains unchanged. This is a separate security control for a different file with different trust semantics.

**Rationale:** `settings.json` hooks can execute code silently on every session start. MCP servers require explicit invocation by the user during a session. The risk profile is different, but still significant — hence opt-in rather than default-on.

### Design

**Allowlist approach** — only extract known-safe keys:
```ts
const ALWAYS_SYNCABLE_KEYS = ["theme", "preferredNotifChannel"];
const OPT_IN_SYNCABLE_KEYS = ["mcpServers"]; // requires claudeJson.syncMcpServers: true
```

Everything else (OAuth, projects, feature flags, usage tracking) is implicitly excluded.

**Store location:** `~/.claudefy/store/config/claude-json-sync.json`

### ClaudeJsonSync class

```ts
export interface ClaudeJsonSyncOptions {
  claudeJsonPath: string;
  storePath: string;
  homeDir: string;           // for @@HOME@@ sentinel
  syncMcpServers: boolean;   // from config
}

export class ClaudeJsonSync {
  /**
   * Extract syncable fields from ~/.claude.json.
   * Returns the extracted object for the encryption pipeline.
   */
  extract(options: ClaudeJsonSyncOptions): Record<string, unknown>;

  /**
   * Merge stored fields into local ~/.claude.json.
   * Returns the merged full object — caller writes to disk.
   */
  merge(options: ClaudeJsonSyncOptions): Record<string, unknown>;
}
```

Note: `merge()` returns the merged object instead of writing directly. This follows the immutability convention (global coding-style.md) — the caller handles the disk write with proper atomic semantics.

### MCP server merge strategy

- Remote-only server → added to local
- Local-only server → preserved (no deletions)
- Both have same server name → remote config wins
- Union semantics: both machines end up with all servers

### Path mapping

**Dedicated sentinel handling** — `ClaudeJsonSync` handles `@@HOME@@` expansion itself, without delegating to `PathMapper.remapPathField()`. This avoids conflict with `PathMapper`'s existing `@@alias@@` regex, which would try to look up `"HOME"` in the links map and fail silently.

```ts
// Extract (push): replace home dir with sentinel
private canonicalizePaths(obj: unknown, homeDir: string): unknown {
  // Deep-walk strings, replace homeDir prefix with @@HOME@@
}

// Merge (pull): replace sentinel with local home dir
private localizePaths(obj: unknown, homeDir: string): unknown {
  // Deep-walk strings, replace @@HOME@@ with homeDir
}
```

### Config

```ts
claudeJson: {
  sync: true,              // master switch
  syncMcpServers: false,   // opt-in for MCP servers (security-sensitive)
},
```

### Push integration

In `push.ts`, after the main store sync completes:

```ts
// After existing syncItem() loop:
if (config.claudeJson?.sync !== false) {
  const claudeJsonPath = join(homeDir, ".claude.json");
  if (existsSync(claudeJsonPath)) {
    const sync = new ClaudeJsonSync();
    const extracted = sync.extract({ claudeJsonPath, storePath, homeDir, syncMcpServers });
    // Write extracted data to store/config/claude-json-sync.json
    writeFileSync(storePath, JSON.stringify(extracted, null, 2));
    // Add to changedFiles for secret scanning
    changedFiles.push(storePath);
  }
}
```

**Key detail:** `claude-json-sync.json` is added to `changedFiles` so the secret scanner processes it. If scanner finds secrets (e.g., API keys in MCP server args), the file gets encrypted per the reactive/full encryption model. Without this integration, MCP configs containing secrets would be stored in plaintext.

### Pull integration

In `pull.ts`, inside the main `try` block (after step 6 — settings merge, before the final file copy loop):

```ts
// Step 7: ~/.claude.json merge
if (config.claudeJson?.sync !== false) {
  const storeFile = join(tmpConfigDir, "claude-json-sync.json");
  if (existsSync(storeFile)) {
    const sync = new ClaudeJsonSync();
    const merged = sync.merge({ claudeJsonPath, storePath: storeFile, homeDir, syncMcpServers });
    // Write to temp file, then atomic rename
    const tmpOut = join(tmpDir, "claude-json-merged.json");
    writeFileSync(tmpOut, JSON.stringify(merged, null, 2));
    // Copy permissions from original if it exists
    if (existsSync(claudeJsonPath)) {
      const stat = statSync(claudeJsonPath);
      chmodSync(tmpOut, stat.mode);
    }
    renameSync(tmpOut, claudeJsonPath);
  }
}
```

**Key detail:** The merge happens inside the `try` block, within the signal-interruption checkpoint system. If `interrupted` is true, this step is skipped (same as file copy operations). The temp file is written inside `tmpDir` so it gets cleaned up by the `finally` block if the operation is aborted.

### Safety

- Missing `~/.claude.json` (local) → skip silently on both push and pull
- Missing `claude-json-sync.json` (store) → skip silently on pull. Local `~/.claude.json` is left unchanged. This is the "no remote data" case — e.g., first pull on a new machine joining a store that hasn't pushed claude.json yet.
- Malformed JSON → log warning, skip
- Atomic writes: write to temp file in `tmpDir`, copy permissions, then `fs.rename()`
- Secret scanner runs against `claude-json-sync.json` via `changedFiles` integration

### Tests

- Extract only syncable keys from full fixture
- OAuth/project fields never in extracted output
- `syncMcpServers: false` → `mcpServers` excluded from extraction
- `syncMcpServers: true` → `mcpServers` included
- Merge: remote server added to local
- Merge: local-only server preserved (no deletion)
- Merge: overlapping server name → remote config wins
- Merge: stdio server (`{ command, args }`) vs SSE server (`{ url }`) → handles both types
- Path mapping: `/Users/jose/.local/bin/server` → `@@HOME@@/.local/bin/server` round-trip
- Path mapping: home dir with spaces → correct round-trip
- `@@HOME@@` sentinel NOT processed by PathMapper (isolation test)
- Missing `~/.claude.json` (local) → skip without error
- Missing `claude-json-sync.json` (store) → skip, local file unchanged
- Malformed JSON → warning, skip without crash
- `claudeJson.sync: false` → entire feature skipped
- File permissions preserved on write-back
- Secret scanner runs against `claude-json-sync.json` when present in `changedFiles`
- Encryption: `claude-json-sync.json` encrypted in `"full"` mode
- Encryption: `claude-json-sync.json` encrypted in `"reactive"` mode when scanner finds secrets in it
- SIGINT during pull → `claude-json-sync` merge skipped if interrupted, temp files cleaned up
